# TDD Strategy: Overlayer

## Philosophy: Outside-In TDD

The strategy follows an outside-in rhythm (Acceptance → Integration → Unit):

1. **Acceptance** tests define the system's contract and overall orchestration. They are the first tests written.
2. **Integration** tests verify boundaries with external infrastructure (e.g., AWS, FFmpeg) that cannot be safely mocked.
3. **Unit** tests cover complex, isolated domain logic, state branching, and algorithms without infrastructure overhead.

This approach ensures the system meets its high-level requirements while maintaining fast feedback loops for complex logic.

## 1. Overlayer.Api (.NET Minimal API)

The API layer is primarily responsible for HTTP contracts, request validation, and generating infrastructure instructions (e.g., presigned URLs).

- **Acceptance:** Uses `WebApplicationFactory` to validate the HTTP contract from the outside. These tests ensure the API behaves correctly against the documented contract, validating shapes, headers, and routing.
- **Integration:** Tests the API against real infrastructure emulators (like LocalStack S3) when AWS wire compatibility is essential. This validates that the presigned URLs and configuration generated by the API actually work in practice.
- **Unit:** Isolates specific business logic (like URL generation rules, status inference, and error handling) from the HTTP pipeline and infrastructure. This allows for exhaustive testing of edge cases (e.g., MIME types, file sizes) using mocked dependencies.

## 2. Overlayer.Worker (.NET Console App)

The worker orchestrates the pipeline, coordinates distributed state, and executes heavy processing.

- **Acceptance:** Validates the message consumption and orchestration loop. Using mocked queues and processors, these tests ensure the worker correctly pulls work, routes it, and acknowledges completion without getting bogged down in actual video rendering.
- **Integration:** Verifies the actual execution pipeline against emulated infrastructure (LocalStack) and real binaries (FFmpeg). This layer proves that the worker can successfully interact with S3 and execute external processes. Additional integration tests cover SQS queue semantics (polling, idempotency).
- **Unit:** Tests the isolated decision-making logic of the worker without external dependencies. This includes pipeline branching, idempotency checks, S3 lock strategy logic (ADR 008), command building for FFmpeg, and failure handling (ADR 007).
